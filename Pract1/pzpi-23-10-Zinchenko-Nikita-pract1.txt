3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони (патерни) проєктування - це типові, перевірені часом рішення поширених проблем, що виникають під час проєктування архітектури програмного забезпечення. Згідно з класифікацією "Банди чотирьох" (GoF), шаблони поділяються на три основні групи:
     1. Твірні (породжувальні): відповідають за гнучке створення об'єктів без внесення зайвих залежностей (наприклад, Builder, Factory Method).
     2. Структурні: допомагають збирати об'єкти та класи у більші структури, зберігаючи їхню гнучкість та ефективність (наприклад, Adapter, Facade).
     3. Поведінкові: визначають алгоритми та способи взаємодії між об'єктами (наприклад, Observer, Strategy). Область застосування патернів охоплює всі етапи розробки програмного забезпечення, де необхідно забезпечити масштабованість, читабельність коду та легкість його подальшої підтримки.
3.2 Опис обраного патерна
     Назва патерна: Builder (Будівельник)
     Тип шаблона: Твірний (породжувальний)
     Основне призначення: Шаблон розділяє процес конструювання складного об'єкта від його представлення. Це дозволяє використовувати один і той самий процес конструювання для створення різних конфігурацій об'єктів. Ефективно вирішує проблему "телескопічного конструктора" (коли клас має багато перевантажених конструкторів з різною кількістю параметрів).
     Передумови застосування: Використання є доцільним, коли об'єкт має складну структуру, велику кількість властивостей (багато з яких є необов'язковими), або коли процес створення об'єкта повинен складатися з певних послідовних кроків (наприклад, збірка складного HTTP-запиту, конфігурація UI-елементів).
     
     Переваги: 
     Дозволяє створювати об'єкти покроково.
     Ізолює складний код збірки об'єкта від його основної бізнес-логіки.
     Надає повний контроль над процесом конструювання.
     Недоліки:
     Збільшує загальну кількість класів та обсяг коду в проєкті.
     Вимагає створення окремого класу-будівельника для кожного типу продукту (у класичній реалізації GoF).
     
3.3 Структура шаблона (патерна)
Опис структурних частин (Учасники):
Builder (Будівельник):
Задає абстрактний інтерфейс для створення окремих частин складного об'єкта Product.
ConcreteBuilder (Конкретний будівельник):
Конструює та збирає разом частини продукту шляхом реалізації інтерфейсу Builder.
Визначає внутрішнє представлення об'єкта, що створюється, і відстежує процес його збірки.
Надає інтерфейс для доступу до готового продукту (метод GetResult()).
Director (Керівник / Директор):
Конструює об'єкт, використовуючи виключно абстрактний інтерфейс Builder. Він знає послідовність кроків, але не знає деталей їх реалізації.
Product (Продукт):
Представляє складний конструйований об'єкт. Внутрішнє представлення продукту та процес його остаточної збірки визначаються класом ConcreteBuilder.
Включає класи, які визначають складові частини продукту, а також інтерфейси для збірки кінцевого результату з цих частин.
     Опис взаємодії об’єктів (Відносини): Алгоритм взаємодії між учасниками патерна виглядає наступним чином:
1. Клієнтський код створює об'єкт-керівник (Director) та конфігурує його потрібним конкретним об'єктом-будівельником (ConcreteBuilder).
2. Керівник послідовно повідомляє будівельника про те, що потрібно побудувати чергову частину продукту (через виклик методів, наприклад, BuildPart()).
3. Будівельник обробляє ці запити від керівника, виконує необхідну логіку та крок за кроком додає нові частини до продукту, що конструюється.
4. Після завершення процесу конструювання, клієнт забирає повністю готовий продукт безпосередньо у будівельника.

     3.4 Реалізація шаблона (патерна)
     Опис реалізації мовою програмування: C++ (на основі класичного прикладу побудови лабіринту з книги GoF).
     Основні етапи:
     1. Створення базових інтерфейсів Створюється абстрактний базовий клас MazeBuilder. У C++ його методи навмисно зроблені не суто віртуальними (pure virtual), а порожніми. Це дозволяє підкласам-спадкоємцям перевизначати лише ті операції конструювання, які їм дійсно потрібні, не реалізовуючи зайвий код.
C++
class Maze; // Клас Продукту

class MazeBuilder {
public:
    virtual void BuildMaze() { }
    virtual void BuildRoom(int room) { }
    virtual void BuildDoor(int roomFrom, int roomTo) { }
    
    virtual Maze* GetMaze() { 
return nullptr; 
}

protected:
    MazeBuilder() {}
};
     2. Реалізація конкретних класів Створюється конкретний будівельник StandardMazeBuilder, який успадковує MazeBuilder. Він містить реальну логіку створення кімнат, стін та дверей, а також зберігає стан поточного лабіринту під час його збірки.
class StandardMazeBuilder : public MazeBuilder {
private:
    Maze* _currentMaze; // Внутрішнє представлення продукту

public:
    StandardMazeBuilder() { 
_currentMaze = nullptr; 
}

    virtual void BuildMaze() override { 
        _currentMaze = new Maze(); 
    }
    
    virtual void BuildRoom(int n) override {
        // логіка створення кімнати та стін навколо неї
    }
    
    virtual void BuildDoor(int n1, int n2) override {
        // логіка створення дверей між двома кімнатами
    }
    
    virtual Maze* GetMaze() override { 
        return _currentMaze; 
    }
};

     3. Реалізація Керівника (Director) Клас MazeGame виступає у ролі Директора. Він має метод CreateMaze, який приймає будь-якого будівельника через абстрактний інтерфейс і задає чітку послідовність кроків для конструювання. Керівник нічого не знає про те, як саме будуються стіни чи двері.

class MazeGame {
public:
    Maze* CreateMaze(MazeBuilder& builder) {
        builder.BuildMaze();
        builder.BuildRoom(1);
        builder.BuildRoom(2);
        builder.BuildDoor(1, 2);
        
        return builder.GetMaze();
    }
};

     4. Демонстрація використання У клієнтському коді створюються об'єкти Директора та конкретного Будівельника. Клієнт передає будівельника директору для виконання роботи, а після завершення - забирає готовий лабіринт.

int main() {
    MazeGame game;                 // Створюємо Керівника
    StandardMazeBuilder builder;   // Створюємо Конкретного Будівельника

    // Керівник покроково конструює лабіринт
    game.CreateMaze(builder);

    // Клієнт забирає готовий продукт у Будівельника
    Maze* maze = builder.GetMaze();

    return 0;
}

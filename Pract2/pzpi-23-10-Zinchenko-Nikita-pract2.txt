3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про систему
     Назва системи: Glovo.
     Розробник: Glovoapp 23SL.
     Призначення: Багатокатегорійна платформа доставки «на вимогу». Користувачі: клієнти (замовники), кур'єри та партнери (ресторани, магазини).
     
     3.2 Архітектурний стиль системи
     Стиль: Мікросервісна архітектура, керована подіями (Event-Driven Architecture).
     Обґрунтування: Система обробляє тисячі замовлень одночасно в різних країнах. Мікросервіси дозволяють окремим командам розробляти та масштабувати сервіс оплати, сервіс логістики та сервіс каталогів незалежно один від одного. Подієва модель (через Apache Kafka) забезпечує реакцію системи на зміну статусу замовлення в реальному часі.

     3.3 Основні компоненти системи
     Опис підсистем:
     Glovo App: Інтерфейс для пошуку товарів та оформлення замовлень.
     Glovo Rider: Додаток для керування замовленнями, навігації та відстеження доходів.
     Order Management System (OMS): Центральний сервіс, що керує життєвим циклом замовлення.
     Logistics & Dispatching Engine: «Мозок» системи, який за допомогою алгоритмів машинного навчання призначає замовлення найближчому доступному кур'єру.
     Payment Gateway: Обробка транзакцій та виплат партнерам і кур'єрам.

     3.4 Взаємодія компонентів
     Взаємодія підсистем у платформі базується на подієво-орієнтованій архітектурі та використанні брокерів повідомлень для асинхронного зв'язку. Для ілюстрації цих процесів розроблено три UML-діаграми.
     UML-діаграма потоків даних (Data Flow / Activity Diagram) На рисунку 1.1 зображено потік даних геопозиції кур'єра. Дані (Location Data Object, що містить широту, довготу та час) генеруються на мобільному пристрої кур'єра (Courier App) і відправляються на сервер. API Gateway виконує валідацію запиту та перетворює дані у формат GEOJSON. Після цього бекенд оновлює геопросторові індекси у швидкій in-memory базі даних Redis за допомогою команди GEOADD. Це дозволяє системі миттєво знаходити кур'єрів поблизу.

Рисунок 1.1 - Data Flow / Activity Diagram
     
     UML-діаграма протоколів взаємодії (Sequence Diagram) На рисунку 1.2 показано процес призначення замовлення. Клієнт створює замовлення через синхронний REST-запит до Order Service. Сервіс відповідає статусом створення і генерує асинхронну подію OrderCreated у Message Broker. Підсистема диспетчеризації (Dispatch Engine) споживає цю подію, знаходить найближчого кур'єра та відправляє йому Push-сповіщення. Кур'єр підтверджує прийняття замовлення через POST-запит до API, на що система відповідає статусом 200 OK.

     3.5 Масштабованість та надійність
     Контейнеризація: Використання Docker та Kubernetes для автоматичного масштабування під час пікових годин.
     Кешування: Redis для швидкого доступу до статусів кур'єрів та популярних меню.
     Бази даних: PostgreSQL для транзакційних даних та DynamoDB для логів і аналітики.
     3.6 Приклади програмної реалізації
     Для демонстрації архітектурних принципів роботи системи було використано запити до сервісів ШІ. Згенеровані фрагменти коду (наведено у Додатку В) написані на C# (.NET) та демонструють ключові аспекти функціонування бекенду:
     1. Алгоритм диспетчеризації та API: Реалізація DispatchEngine показує роботу бізнес-логіки з пошуку найближчого кур'єра з використанням LINQ-запитів до репозиторію. CourierController ілюструє обробку HTTP-запитів REST API для прийняття замовлень.
     2. Робота з JSON: Продемонстровано модель даних LocationUpdate та приклад JSON-документа, за допомогою якого мобільний додаток кур'єра передає свої GPS-координати на сервер.
     3. Робота з файлами: Написано клас DispatchLogger, який використовує асинхронні потоки для запису логів помилок диспетчеризації у текстовий файл. Це забезпечує базовий рівень моніторингу роботи мікросервісу.

1  // 1. HTTP-запит до API та бізнес-логіка (C#)
2  public class DispatchEngine : IDispatchEngine
3  {
4      private readonly ICourierRepository _courierRepo;
5  
6      public DispatchEngine(ICourierRepository courierRepo)
7      {
8          _courierRepo = courierRepo;
9      }
10 
11     public Courier AssignOrderToNearestCourier(Order order)
12     {
13         var availableCouriers = _courierRepo.GetAvailableCouriers();
14         var nearestCourier = availableCouriers
15             .OrderBy(c => CalculateDistance(c.Location, order.RestaurantLocation))
16             .FirstOrDefault();
17             
18         if (nearestCourier != null)
19         {
20             nearestCourier.AssignedOrderId = order.Id;
21             _courierRepo.UpdateStatus(nearestCourier.Id, CourierStatus.Busy);
22         }
23         return nearestCourier;
24     }
25 
26     private double CalculateDistance(Location loc1, Location loc2)
27     {
28         return Math.Sqrt(Math.Pow(loc1.Latitude - loc2.Latitude, 2) + 
29                          Math.Pow(loc1.Longitude - loc2.Longitude, 2));
30     }
31 }
